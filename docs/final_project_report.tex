\begin{abstract}
This report documents the complete lifecycle of the \textbf{Sentiment Analysis Service}, a production-grade ML system developed as the capstone project for the DDM501 course.
The system performs real-time sentence-level sentiment classification (positive / negative / neutral) as well as Aspect-Based Sentiment Analysis (ABSA). The core model is \texttt{xlm-roberta-base}, fine-tuned via Parameter-Efficient Fine-Tuning (PEFT/LoRA) to produce separate lightweight adapters for Sentiment Analysis and Sarcasm Detection, enabling multilingual support across English and Vietnamese.

The service is exposed through a RESTful FastAPI application, containerised with Docker, orchestrated via Docker Compose, and equipped with a full observability stack (Prometheus + Grafana), experiment tracking (MLflow), automated data validation (DVC), a robust test suite, and a GitHub Actions CI/CD pipeline.
A dedicated \texttt{/explain} endpoint delivers token-level SHAP attributions, fulfilling the Responsible AI explainability requirement.
\end{abstract}

\subsection{Problem Statement \& Business Context}

Customer feedback represents one of the most valuable yet under-utilised assets of any business.
Manual review of large volumes of reviews, social-media posts, and support tickets is both expensive and slow.

The \textbf{Sentiment Analysis Service} addresses this gap by providing an automated, REST-accessible system that:
\begin{enumerate}[label=\alph*)]
  \item classifies the \emph{overall} sentiment of a text snippet (positive · negative · neutral);
  \item extracts \emph{aspect-level} sentiments (food, service, ambiance, price, location, general) for fine-grained business insight;
  \item provides \emph{SHAP-based token explanations} so that end-users can trust and audit model decisions.
\end{enumerate}

The primary dataset used for offline evaluation is the \textbf{SemEval-2014 Task 4 Restaurant Reviews} corpus, which provides sentence-level and aspect-level sentiment annotations.

\subsection{Data Flow}

\begin{enumerate}
  \item User submits text to \texttt{POST /predict} (or \texttt{/explain}) via the Angular frontend or any HTTP client.
  \item FastAPI validates the request body through Pydantic schemas (\texttt{PredictRequest}).
  \item Validated text is forwarded to \texttt{BaselineModelInference\allowbreak.predict\_single()}.
  \item The XLM-RoBERTa tokenizer encodes the text; the ONNX-optimised model with merged LoRA weights returns a softmax output giving sentiment label and confidence.
  \item The ABSA module extracts aspect-level sentiments (food, service, ambiance, price, location, general) above a 0.5 threshold.
  \item For \texttt{/explain}, a SHAP \texttt{Explainer} returns per-token attribution scores.
  \item The response is serialised by \texttt{PredictResponse} and returned to the client.
  \item The monitoring middleware records request counts and latencies for Prometheus.
\end{enumerate}

\subsection{ML Pipeline (15\%)}

\subsubsection{Datasets \& Machine Learning Models}

The system integrates diverse data sources and advanced modelling techniques to support multiple text-classification tasks:

\begin{itemize}[noitemsep]
  \item \textbf{Datasets:}
  \begin{itemize}[noitemsep]
    \item \textbf{ABSA SemEval 2014 (Restaurants):} Provides sentence and aspect-level sentiment annotations. Used for baseline evaluation.
    \item \textbf{Sarcasm Dataset (\texttt{tweet\_eval\_irony\_v1}):} Identifies irony and sarcasm in social media contexts.
    \item \textbf{Multilingual Sentiment (\texttt{multilingual\_sentiment\_v1}):} English and Vietnamese data for training multilingual capabilities.
  \end{itemize}
  \item \textbf{Machine Learning Models:}
  \begin{itemize}[noitemsep]
    \item \textbf{Base Model (\texttt{xlm-roberta-base}):} Serves as a unified backbone for superior cross-lingual understanding.
    \item \textbf{Training Strategy (PEFT/LoRA):} Parameter-Efficient Fine-Tuning via Low-Rank Adaptation (LoRA) is employed. Separate lightweight adapters are trained for Sentiment Analysis and Sarcasm Detection, allowing multiple adapters to be stacked efficiently during inference to minimise memory footprint.
    \item \textbf{Hardware-Aware Optimisation:} The training pipeline automatically detects available accelerators (CUDA, MPS, CPU). On supported GPUs (e.g., L4), it automatically enables Mixed Precision (\texttt{bf16}), fused optimisers (\texttt{adamw\_torch\_fused}), pinned memory, and tuned Data Loader workers to maximise training throughput.
    \item \textbf{Inference Optimisation (ONNX \& Quantization):} HuggingFace models are exported to ONNX format (both FP32 and INT8 quantization), significantly reducing inference latency and resource consumption in production.
  \end{itemize}
\end{itemize}

\subsubsection{Reproducible DVC Pipeline}

The complete lifecycle of data and models is orchestrated via Data Version Control (DVC). The pipeline is defined as a Directed Acyclic Graph (DAG) in \texttt{dvc.yaml} to ensure strict reproducibility:

\begin{enumerate}[noitemsep]
  \item \textbf{Data Engineering:}
  \begin{itemize}[noitemsep]
    \item \texttt{download}: Automatically retrieves raw data from external sources.
    \item \texttt{preprocess}: Executes a configurable transformation chain (label mapping, sentiment derivation, cleaning, deduplication, length filtering, and Train/Test splitting according to \texttt{params.yaml}).
    \item \texttt{validate}: Enforces data quality checks (null ratios, minimum samples, label distributions) and automatically generates \texttt{quality\_report.json}.
  \end{itemize}
  \item \textbf{Modeling:}
  \begin{itemize}[noitemsep]
    \item \texttt{download\_sarcasm} / \texttt{download\_sentiment}: Prepares task-specific training datasets.
    \item \texttt{finetune\_sarcasm} / \texttt{finetune\_sentiment}: Triggers LoRA adapter training.
    \item \texttt{evaluate\_baseline} / \texttt{evaluate\_finetuned}: Evaluates models on test splits, generating detailed reports: \texttt{metrics\_finetuned.json}, \texttt{per\_language\_f1.json}, and \texttt{fairness\_report.json}.
  \end{itemize}
  \item \textbf{Deployment Preparation:}
  \begin{itemize}[noitemsep]
    \item \texttt{export\_onnx\_sentiment} / \texttt{export\_onnx\_sarcasm}: Merges LoRA weights into the base model and exports ONNX graphs.
    \item \texttt{benchmark\_onnx}: Measures real-world inference performance, yielding \texttt{onnx\_benchmark.json}.
  \end{itemize}
\end{enumerate}

\subsubsection{Experiment Tracking with MLflow}

MLflow is deeply integrated into the source code to track the entire process from data validation to model training:

\begin{itemize}[noitemsep]
  \item \textbf{Data Quality Tracking:} The Data Validation process automatically logs metrics directly to the \texttt{data\_preprocessing} experiment. Metrics include \texttt{total\_samples}, split-specific null ratios, and a \texttt{passed\_quality\_checks} flag. The \texttt{quality\_report.json} is attached as an artifact.
  \item \textbf{Model Training Tracking:} 
  \begin{itemize}[noitemsep]
    \item Integration with HuggingFace \texttt{Trainer} (\texttt{report\_to=["mlflow"]}) automatically logs loss, learning rate, and metrics per epoch.
    \item \textbf{Organisational Structure:} Runs are organised into task-specific experiments (e.g., \texttt{phase2\_finetuning\_sentiment}, \texttt{phase2\_finetuning\_sarcasm}).
    \item \textbf{Automatic Tagging:} Each run is explicitly tagged with \texttt{task}, \texttt{git\_sha} (linking model version to code), \texttt{device}, and \texttt{dataset\_version}.
    \item \textbf{Hyperparameter Tracking:} Parameters such as \texttt{learning\_rate}, \texttt{batch\_size}, and \texttt{epochs} are automatically ingested from the centralised \texttt{params.yaml} and logged for comparative analysis.
  \end{itemize}
\end{itemize}

\textbf{Error Handling:}~Custom exception handlers return appropriate HTTP status codes:
\begin{itemize}[noitemsep]
  \item \texttt{UnsupportedLanguageError} $\rightarrow$ HTTP 400 plain-text message.
  \item \texttt{ModelError} $\rightarrow$ HTTP 500 plain-text message.
  \item Model not yet loaded $\rightarrow$ HTTP 503 via the \texttt{get\_model} dependency.
\end{itemize}

Request and response schemas are enforced via Pydantic v2 with field-level validation:

\begin{lstlisting}[language=Python, caption={Core Pydantic schemas (contracts/schemas.py)}]
class PredictRequest(BaseModel):
    text: str = Field(min_length=1, max_length=2000)
    lang: str = "en"

class PredictResponse(BaseModel):
    text: str
    sentiment: Literal["positive", "negative", "neutral"]
    confidence: float = Field(ge=0.0, le=1.0)
    aspects: list[AspectSentimentOut] = Field(default_factory=list)
    sarcasm_flag: bool
    latency_ms: float

class ExplainResponse(BaseModel):
    tokens: list[str]
    shap_values: list[float]
    base_value: float
    latency_ms: float

    @model_validator(mode="after")
    def validate_lengths(self):
        if len(self.tokens) != len(self.shap_values):
            raise ValueError("tokens and shap_values must match in length")
        return self
\end{lstlisting}

The Dockerfile employs a \textbf{multi-stage build} to minimise the final image size:

\begin{itemize}[noitemsep]
  \item \textbf{Stage 1 (builder):} \texttt{python:3.11-slim} with \texttt{build-essential}; installs PyTorch (CPU wheel) and all \texttt{requirements.txt} packages.
  \item \textbf{Stage 2 (runtime):} copies only \texttt{site-packages} and \texttt{bin} from the builder, plus application source.
  \item \textbf{Model caching:} \texttt{RUN python src/model/download\_models.py} bakes the HuggingFace weights into the image at build time.
  \item \textbf{Offline enforcement:} \texttt{TRANSFORMERS\_OFFLINE=1} and \texttt{HF\_DATASETS\_OFFLINE=1} eliminate network calls at runtime.
\end{itemize}

\begin{lstlisting}[language=bash, caption={Dockerfile multi-stage build summary}]
# Stage 1: build
FROM python:3.11-slim AS builder
RUN pip install torch --index-url https://download.pytorch.org/whl/cpu \
 && pip install -r requirements.txt

# Stage 2: lean runtime
FROM python:3.11-slim
COPY --from=builder /usr/local/lib/python3.11/site-packages ...
COPY src/ contracts/
RUN python src/model/download_models.py
ENV TRANSFORMERS_OFFLINE=1
CMD ["uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8000"]
\end{lstlisting}

\subsection{Model Explainability with SHAP}

The \texttt{/explain} endpoint exposes token-level feature attributions using the \textbf{SHAP} library (\texttt{shap\,$\geq$\,0.42.0}).

\subsubsection{Technical Implementation}

\begin{lstlisting}[language=Python, caption={SHAP explanation in BaselineModelInference}]
def get_shap_explanation(self, text: str, lang: str = "en") -> SHAPResult:
    import shap
    self._check_language(lang)
    # Identify the predicted class index
    predicted_probs     = self._predict_probabilities(text)[0]
    predicted_class_idx = int(predicted_probs.argmax().item())
    # Wrap HuggingFace pipeline in SHAP Explainer
    pipe        = self._get_classification_pipeline()
    explainer   = shap.Explainer(pipe)
    shap_values = explainer([text])
    # Extract token strings and SHAP scores for the predicted class
    raw_tokens = shap_values.data[0]
    tokens = raw_tokens.tolist() if hasattr(raw_tokens, "tolist") else list(raw_tokens)
    values = shap_values.values[0][:, predicted_class_idx].tolist()
    base   = float(shap_values.base_values[0][predicted_class_idx])
    return SHAPResult(tokens=tokens, shap_values=values, base_value=base)
\end{lstlisting}

The explanation workflow:
\begin{enumerate}[noitemsep]
  \item The text is classified to determine the predicted class index.
  \item \texttt{shap.Explainer} computes Shapley values by marginalising over permutations of input tokens.
  \item Per-token SHAP scores for the predicted class are extracted alongside the base value.
  \item The \texttt{ExplainResponse} schema enforces \texttt{len(tokens) == len(shap\_values)} via \texttt{model\_validator}.
\end{enumerate}

\subsection{Fairness Analysis}

\textbf{Known Biases.}~The \texttt{xlm-roberta-base} model, fine-tuned via LoRA on the \texttt{multilingual\_sentiment\_v1} dataset, may exhibit the following biases:
\begin{itemize}[noitemsep]
  \item \textbf{Domain bias:} Training data skews toward social-media and review text; performance may degrade on highly formal or technical language.
  \item \textbf{Language imbalance:} English training samples may outnumber Vietnamese samples, potentially reducing fairness across languages. Per-language F1 scores are tracked in \texttt{per\_language\_f1.json} and logged to MLflow.
  \item \textbf{Sarcasm:} The dedicated LoRA sarcasm adapter (trained on \texttt{tweet\_eval\_irony\_v1}) addresses this limitation; the \texttt{sarcasm\_flag} field in the API response is populated accordingly.
\end{itemize}

\textbf{Dataset Bias Mitigation.}~The \texttt{DataQualityValidator} emits distribution warnings when any class falls below 5\% in a split, enabling early detection of imbalance before evaluation.

\subsection{Data Privacy Considerations}

\begin{itemize}[noitemsep]
  \item No user input data is persisted; the service is \textbf{stateless} -- all processing is in-memory per request.
  \item No PII is logged by the monitoring middleware; only HTTP method, path, status code, and timing are recorded.
  \item Model weights are stored inside the Docker image; no external registry calls occur at runtime.
\end{itemize}

\section{Conclusion}

The Sentiment Analysis Service successfully demonstrates a complete, production-grade ML system lifecycle:

\begin{enumerate}
  \item \textbf{Problem Definition:} Clearly motivated by business need; success metrics defined at business, model, and system levels.
  \item \textbf{Architecture:} Microservices design with full observability and experiment tracking, all resource-limited via Docker Compose.
  \item \textbf{ML Pipeline:} Reproducible data pipeline (DVC) encompassing three datasets (SemEval-2014, \texttt{tweet\_eval\_irony\_v1}, \texttt{multilingual\_sentiment\_v1}), data-quality validation, PEFT/LoRA fine-tuning of \texttt{xlm-roberta-base} for Sentiment and Sarcasm tasks, ONNX export with INT8 quantization, and MLflow-tracked evaluations including per-language fairness reporting.

  \item \textbf{Monitoring:} Three Prometheus metrics via ASGI middleware; Grafana dashboards auto-provisioned; two alerting rules for error rate and inference latency.
  \item \textbf{Testing \& CI/CD:} Three test layers (unit, integration, data quality); GitHub Actions lints and tests on every commit.
  \item \textbf{Responsible AI:} SHAP token attributions via \texttt{/explain}; per-language fairness analysis; privacy-preserving stateless design; ethical risk register.
\end{enumerate}
